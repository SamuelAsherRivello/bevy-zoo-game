# Research: Project Setup

| Decision | Rationale | Alternatives Considered |
| -------- | --------- | ----------------------- |
| Use `scripts/other/CompileApp.ps1` for headless Cargo work | Desktop, web, tests, and dependency warmup share target-dir, feature, target triple, linker, and incremental setup instead of duplicating compile logic | Keeping separate Cargo command construction in each script was rejected because the scripts had started to drift |
| Add a dedicated desktop hot reload script | `bevy-jam-1` uses Dioxus CLI hot patching for native Bevy hot reload; keeping this as its own entry point preserves the existing normal desktop script and makes the hot reload dependency explicit | Folding hot reload into `RunAppDesktop.ps1` was rejected because release-like review runs and hot patch sessions have different process lifetimes, tooling requirements, and compatibility constraints |
| Use Dioxus CLI 0.7.x for hot reload tooling | The reference workflow gates on 0.7.x support for hot patching, so dependency setup should verify or install a compatible CLI | Relying on any `dx` version was rejected because unsupported versions may accept different arguments or fail after a long build |
| Disable Bevy dynamic linking by default during Windows hot reload | The reference workflow documents Windows/MSVC conflicts between Bevy dynamic linking and Dioxus hot patch linking; this repo keeps `fast-dev` for normal desktop runs but avoids it in hot reload unless explicitly enabled on a compatible host | Always enabling `fast-dev` was rejected because it can make hot patching less reliable on the primary Windows environment |
| Keep window setup and placement behavior in `bevy/crates/shared` | Window defaults, placement persistence, restore validation, and close-save behavior are reusable system-level runtime features | Keeping placement behavior in `bevy/crates/game` was rejected because it is not card-specific |
| Store placement as ignored local JSON | Human-readable local runtime state is easy to inspect and keep out of source control under `data/local_storage/` | Registry or global app data was rejected because project guidance keeps local persistence inside the repo unless explicitly asked otherwise |
| Save placement only on normal close | Matches clarification and avoids unnecessary file writes during every drag or resize | Continuous save was rejected because it contradicts the clarified save timing |
| Restore exact placement when same screen is available | Supports repeated two-screen review workflows with minimal friction | OS automatic placement was rejected because it does not meet exact restore requirements |
| Center primary screen at 1024x768 for invalid/off-screen placement | Gives a predictable visible fallback and satisfies the clarified default | Nearest screen fallback was rejected because the clarified fallback is primary screen centered |
| Use VS Code task entries for integrated terminal output | The user specifically wants command output in VS Code rather than detached popup PowerShell windows | Detached process launch was rejected because output is not visible in the integrated terminal |
